\subsection{Тестовый датасет}

Для воспроизводимой оценки гипотез H1--H3 сформирован корпус из 15~входных
документов, репрезентирующих три основных сценария применения ИИ-методиста
на платформе Adapstory (приложение~Ж):

\begin{itemize}
  \item 5~документов типа «продуктовое описание» --- маркетинговые брифы,
    описания функциональности сервисов, технические паспорта продуктов
    (объём 1--5~страниц);
  \item 5~документов типа «технический регламент» --- корпоративные
    инструкции, политики безопасности, процессные регламенты
    (объём 3--15~страниц);
  \item 5~документов типа «учебная программа» --- ФГОС-совместимые
    рабочие программы дисциплин, корпоративные учебные планы
    (объём 5--20~страниц).
\end{itemize}

Эталонный корпус (ground~truth) формировался по~двухуровневой схеме
для~снижения субъективности разметки:

\begin{enumerate}
  \item \textbf{Экспертная разметка.}
    Два независимых эксперта-методиста (стаж разработки eLearning-курсов
    $\geq$~5~лет у~каждого) независимо составляли эталонную структуру
    курса по~каждому документу с~разметкой учебных целей по~уровням
    таксономии Блума~\cite{bloom_taxonomy_1956}.
    Согласованность экспертов оценивалась коэффициентом Коэна~$\kappa$
    по~шести уровням Блума; минимально допустимое значение~---
    $\kappa \geq 0.61$ (существенное согласие по~шкале~Landis and Koch~\cite{landis_koch_kappa_1977}).
    При~расхождении (метка одного уровня) финальное решение принималось
    путём обсуждения; при~расхождении~$\geq$~2~уровней документ исключался
    из~корпуса.

  \item \textbf{Открытые курсы как~независимый эталон.}
    Для~документов типа «учебная программа» в~качестве дополнительного
    ground~truth использованы структуры открытых курсов из~публичных
    репозиториев: MIT~OpenCourseWare~\cite{mit_ocw_2024}
    и~аналогичные открытые программы Coursera.
    Открытые курсы, созданные профессиональными академическими методистами,
    служат независимой точкой сравнения, не~зависящей от~суждений
    привлечённых экспертов.
\end{enumerate}

Итоговый корпус содержит 15~документов с~двумя независимо размеченными
эталонами; рассчитанный коэффициент~$\kappa$~= [PLACEHOLDER]
соответствует [существенному / умеренному] согласию.
Эталоны использовались как ground~truth при расчёте метрик качества.

\subsection{Baseline-конфигурации}

Для обоснованного сравнения определены четыре конфигурации,
обозначаемые B1--B4:

\begin{itemize}
  \item \textbf{B1}~--- ручная разработка структуры курса методистом-экспертом
    (золотой стандарт, эталон качества);
  \item \textbf{B2}~--- GPT-4o / Qwen~2.5-72B с~единственным системным
    промптом без агентской оркестрации (one-shot baseline);
  \item \textbf{B3}~--- Coursebox~AI~--- ближайший коммерческий конкурент
    с заявленной поддержкой автогенерации курсов на основе LLM~\cite{coursebox_ai_2024};
  \item \textbf{B4}~--- ИИ-методист (предлагаемая система): мультиагентный
    конвейер LangGraph~+~Haystack~2.x поверх Qwen~2.5-72B через Ollama.
\end{itemize}

Сравнение B2 vs.~B4 при одинаковой базовой LLM позволяет изолировать
вклад агентской архитектуры (гипотеза~H2) от качества самой модели.
Сравнение B3 vs.~B4 даёт позиционирование относительно рынка.
Сравнение B4 vs.~B1 показывает разрыв между автоматизированной
и~экспертной разработкой.

Метрики оценки:

\begin{itemize}
  \item \textbf{Ragas-метрики}: faithfulness (достоверность утверждений
    относительно исходного документа), answer~relevancy
    (релевантность выходной структуры теме документа),
    context~precision (точность извлечённых фрагментов)~\cite{es_ragas_2023};
  \item \textbf{Bloom's coverage}: доля уровней таксономии Блума,
    представленных в~сгенерированных учебных целях (цель~--- $\geq 70\%$,
    т.\,е. минимум 4~из 6~уровней);
  \item \textbf{Время генерации}: от загрузки документа до финальной
    структуры курса (целевой показатель~--- менее 30~минут);
  \item \textbf{Unit-стоимость}: совокупная стоимость GPU-времени и~API-вызовов
    на один курс в~рублях.
\end{itemize}

\subsection{Фреймворк мультиагентной оркестрации}

Конвейер ИИ-методиста представляет собой ориентированный ациклический граф
(DAG) с~условными переходами, параллельными ветками и~точками остановки
для~ревью владельцем продукта. Выбор фреймворка оркестрации является
ключевым архитектурным решением, поскольку определяет трассируемость,
отказоустойчивость и~поддерживаемость всей системы.

В~ходе исследования рассмотрены пять фреймворков, актуальных по~состоянию
на~начало 2025~года: LangGraph~1.x, AutoGen~0.4, CrewAI~0.9, Agno~1.x
и~OpenAI~Agents~SDK~\cite{langgraph_docs_2025, autogen_04_2024,
crewai_docs_2025, agno_docs_2025, openai_agents_sdk_2025}.
Сравнение проводилось по~семи критериям, критичным для задачи
генерации образовательного контента (таблица~\ref{tab:frameworks}).

\textbf{Обоснование выбора LangGraph.}
Задача ИИ-методиста структурно является stateful~DAG: входной документ
проходит последовательные этапы анализа, структурирования, генерации контента,
проверки фактов и~ревью, причём некоторые ветки выполняются параллельно
(например, генерация иллюстраций и~TTS-подкастов), а~часть переходов
условна (пересмотр структуры при низком Bloom's~coverage).
Именно для таких задач разработан тип \texttt{StateGraph} в~LangGraph~\cite{langgraph_docs_2025}.

Встроенная интеграция с~LangSmith обеспечивает трассировку каждого
вызова LLM на уровне граней агента, что является критически важным
при~отладке качества русскоязычной генерации.
Human-in-the-loop реализован как \texttt{interrupt\_before} / \texttt{interrupt\_after}
на~уровне нод графа~--- first-class~API без обходных решений.
Наконец, зрелость экосистемы (проект LangChain существует с~2022~г.,
LangGraph~--- с~2024~г., стабильный релиз~v1.0 вышел в~начале 2025~г.)
снижает риски технологической нестабильности.

\textbf{Исключённые альтернативы.}
AutoGen~0.4 предлагает мощную actor-модель и~превосходит LangGraph
по~нативному async, однако conversation-centric дизайн менее естественен
для детерминированного производственного конвейера: формирование явного DAG
в~AutoGen требует дополнительной абстракции поверх встроенного API~\cite{autogen_04_2024}.

CrewAI обладает интуитивной ролевой моделью (агент~= роль~+ цель), которая
хорошо документирована, однако в~ходе прототипирования была выявлена
принципиальная несовместимость с~асинхронным стеком
Celery~+~Redis~\cite{crewai_async_issue_2024}: конкурентное выполнение
нод в~Celery-воркерах требует полноценного async~API, которое в~CrewAI~0.9
не~является стабильным.

Agno (ранее~--- Phi-Data) демонстрирует сильный async и~встроенный RAG-модуль,
однако первый стабильный релиз вышел лишь в~конце 2024~года, и~риски
несовместимости API в~production оказались неприемлемы для ВКР с~жёстким
deadline защиты~\cite{agno_docs_2025}.

OpenAI~Agents~SDK исключён на ранней стадии: отсутствие поддержки
локальных LLM (Ollama) противоречит требованию суверенитета данных~---
учебные материалы корпоративных клиентов не~должны покидать периметр
организации.

\subsection{Гибридная архитектура: Haystack~2.x + LangGraph}

После выбора LangGraph как оркестратора агентов возникает вопрос
о~компоненте Document~Ingestion Pipeline: конвертация документов,
чанкинг, векторизация и~загрузка в~Qdrant.

Нативный LangChain предоставляет модули для этих задач, однако
их~тестируемость ограничена: LangChain Document~Loaders и~Text~Splitters
реализованы в~синхронной парадигме и~плохо компонуются в~явные DAG-пайплайны.
Haystack~2.x, напротив, спроектирован именно как компонентный framework
для обработки документов: каждый шаг (конвертер, сплиттер, эмбеддер,
стор)~--- отдельный компонент с~явным контрактом входа/выхода,
а~\texttt{AsyncPipeline} позволяет параллельно обрабатывать несколько
документов~\cite{haystack2_docs_2024}.

Принятое решение~--- \textbf{гибридная архитектура}:
Haystack~2.x \texttt{AsyncPipeline} отвечает за слой Document~Ingestion
(MinIO~$\rightarrow$~OllamaTextEmbedder~$\rightarrow$~SemanticChunker~$\rightarrow$~Qdrant),
LangGraph отвечает за агентную оркестрацию поверх уже проиндексированного
корпуса знаний.
Такое разделение ответственностей упрощает тестирование каждого слоя
независимо и~соответствует принципу единственной ответственности.

\subsection{Embedding-модели и векторное хранилище}

Для поиска релевантных фрагментов из~загруженных документов
(RAG-слой) необходима модель, качественно обрабатывающая русскоязычный
образовательный текст.

Бенчмарк проводился на~подкорпусе из~100~вопросов, сформированных
по~тестовым документам, с~оценкой по~MRR@10 (Mean Reciprocal Rank)
и~recall@5. Рассматривались: \texttt{intfloat/multilingual-e5-large}~\cite{me5_2024},
\texttt{deepvk/USER-bge-m3}~\cite{user_bge_m3_2024} и~\texttt{nomic-embed-text}
(локальная через Ollama).

По результатам бенчмарка выбрана модель
\texttt{deepvk/USER-bge-m3} как показавшая наилучший
recall@5 на корпусе русскоязычных образовательных документов
(результаты в~разделе~\ref{sec:exp-results}).

В~качестве векторного хранилища используется Qdrant~1.9
как единственная production-ready база, уже развёрнутая в~инфраструктуре
Adapstory, с~поддержкой гибридного поиска (dense~+~sparse)~\cite{qdrant_docs_2024}.

\subsection{Стратегии чанкинга}

Размер и~стратегия разбиения документов на~фрагменты (chunk) существенно
влияет на~faithfulness и~context~precision RAG-ответов.
Сравнивались две стратегии:

\begin{itemize}
  \item \textbf{Рекурсивный чанкинг} (\texttt{RecursiveCharacterTextSplitter}):
    фиксированный размер 512~токенов с~перекрытием 64~токена ---
    стандартный подход из~LangChain~\cite{langchain_docs_2024};
  \item \textbf{Семантический чанкинг} (\texttt{SemanticDoubleMergingSplitter}
    из~Haystack~2.x): границы фрагментов определяются по~разрыву
    семантического сходства между предложениями~\cite{haystack2_docs_2024}.
\end{itemize}

context~precision на~[PLACEHOLDER: XX]\,\% по~сравнению с~рекурсивным
при незначительном увеличении времени индексации.
Принято решение использовать семантический чанкинг в~финальной системе.

\subsection{Модели генерации изображений}

Для~иллюстрирования учебного контента рассматривались
DALL-E~3~\cite{dalle3_2023} (API OpenAI) и~FLUX.1-dev~\cite{flux_2024}
(локально через Ollama / Diffusers):

\begin{itemize}
  \item DALL-E~3 обеспечивает высокое качество изображений, однако
    каждое изображение стоит \$0.04--0.12, что при~среднем курсе
    в~20--40~иллюстраций даёт \$0.8--4.8 на~курс --- значимая
    компонента unit-экономики;
  \item FLUX.1-dev при локальном развёртывании снижает стоимость
    до стоимости GPU-времени ($\approx$\,0--5~руб.\ на~изображение),
    при~качестве, приемлемом для~образовательных иллюстраций.
\end{itemize}

Принята \textbf{гибридная стратегия}: FLUX.1-dev для~быстрых черновых
иллюстраций в~пилотных курсах (MVP-сценарий), DALL-E~3 --- как опциональный
провайдер при~требованиях к~фотореалистичному качеству.

\subsection{Финальная конфигурация}

По~результатам экспериментов принята следующая финальная конфигурация
ИИ-методиста:

\begin{itemize}
  \item \textbf{LLM}: Qwen~2.5-72B-Instruct~(Q4\_K\_M) через Ollama;
  \item \textbf{Оркестратор}: LangGraph~1.x (StateGraph);
  \item \textbf{Document Ingestion}: Haystack~2.x AsyncPipeline;
  \item \textbf{Embedding}: \texttt{deepvk/USER-bge-m3};
  \item \textbf{Векторный стор}: Qdrant~1.9, гибридный поиск;
  \item \textbf{Чанкинг}: семантический (SemanticDoubleMergingSplitter);
  \item \textbf{Text-to-image}: FLUX.1-dev (локально);
  \item \textbf{TTS}: Silero~TTS~v4 (локально, MIT).
\end{itemize}

Данная конфигурация обеспечивает полный суверенитет данных
(все компоненты on-premise), предсказуемую unit-стоимость
(только GPU-ресурсы собственного кластера) и~достаточное качество
для подтверждения гипотез H1--H3 (глава~4).
