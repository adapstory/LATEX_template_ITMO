В~данной главе описывается протокол валидации ИИ-методиста,

\section{Протокол валидации}
% ─────────────────────────────────────────────────────────────────

\subsection{Автоматизированная оценка}

Автоматизированная оценка качества выходных артефактов ИИ-методиста
проводится в~трёх измерениях:

\begin{enumerate}
  \item \textbf{Ragas-метрики} (глава~2): faithfulness, answer~relevancy,
    context~precision;
  \item \textbf{Bloom's~coverage}: доля уровней таксономии Блума,
    покрытых в~наборе учебных целей курса; целевой показатель~--- $\geq 70\%$;
  \item \textbf{Структурное качество}: полнота (наличие всех обязательных
    элементов методологии), согласованность учебных целей с~содержанием,
    BLEU/ROUGE относительно эталонов B1.
\end{enumerate}

\subsection{Протокол экспертной оценки}

Для~независимой верификации результатов привлечены 5--8~экспертов-методистов
(критерии отбора: опыт разработки eLearning-курсов~$\geq$~3~лет,
опыт оценки учебных программ).

Каждому эксперту предъявляются четыре обезличенных варианта структуры
одного курса (B1--B4) в~случайном порядке. Оценка проводится
по~рубрике из~5~критериев (шкала 1--5):

\begin{itemize}
  \item соответствие целевой аудитории;
  \item логика и~последовательность модулей;
  \item качество учебных целей (SMART);
  \item покрытие исходного материала;
  \item практическая применимость.
\end{itemize}

Итоговая оценка --- среднее по~5~критериям.
Слепой дизайн исключает предвзятость в~пользу AI-генерации.
Полные таблицы текущих оценочных артефактов приведены в~приложении~И.

\subsection{Экспертная и пользовательская оценка}

Помимо автоматизированных метрик, для~системы подготовлены два контура
качественной проверки.

Первый контур --- \textbf{слепая экспертная оценка}. Для~него
подготовлена рубрика из~5 критериев и~шаблон таблицы сравнения
конфигураций B1--B4. Этот контур должен быть проведён после стабилизации
мультиагентной конфигурации B4, чтобы экспертам предъявлялись уже
не~прототипные, а воспроизводимые результаты.

Второй контур --- \textbf{пользовательский pilot-протокол}. Он
предполагает измерение time-to-draft, субъективной удобства
(SUS~\cite{brooke_sus_1996}) и~готовности рекомендовать решение
(NPS). Дополнительно в~следующую волну pilot-stage закладываются
task-specific self-efficacy и~perceived competence опросы, выполненные
в~формате retrospective pre-post, а~также follow-up на~transfer
of~learning и~goal attainment через 2--4~недели после использования
системы
~\cite{bandura_self_efficacy_2006, pcs_sdt_2026,
kirkpatrick_model_2026, shilts_retrospective_2008,
shogren_gas_2021}. На~момент текущей редакции работы pilot-сценарий
и~вопросы подготовлены, однако полный цикл пользовательской оценки
ещё не~завершён. Это разграничение принципиально: диплом фиксирует уже
доступные технические результаты и~отдельно описывает инструменты
для~следующей волны validation.

\subsection{Сценарий~1: Онбординг-курс (ADDIE)}

\textbf{Входной документ:} продуктовое описание нового SaaS-инструмента,
8~страниц.

\textbf{Ожидаемый результат:} 6-модульный онбординг-курс с~учебными
целями для~новых сотрудников; время генерации менее 30~минут.

\textbf{Ход выполнения:}
\begin{enumerate}
  \item Пользователь загружает PDF в~систему; запускается Document~Ingestion.
  \item После индексации инициируется CourseStructureAgent
    с~методологией ADDIE.
  \item После~генерации структуры система приостанавливается для~human~review;
    пользователь одобряет структуру.
  \item ContentCreatorAgent генерирует тексты уроков; FactCheckAgent
    верифицирует утверждения.
  \item MediaGenerationAgent параллельно генерирует иллюстрации и~подкасты.
  \item Финальный курс публикуется в~LMS Adapstory.
\end{enumerate}

На~текущем этапе количественно измерен не~весь end-to-end pipeline B4,
а~его наиболее воспроизводимый baseline-слой B2. Для~ядрового корпуса
из~16 документов baseline показывает среднее время генерации структуры
4.31~с, среднее покрытие Bloom's taxonomy 61.5\,\%, долю курсов
с~покрытием $\geq 70\,\%$ на~уровне 18.8\,\% и~структурную валидность
100\,\%. Эти цифры ещё не~являются итоговыми показателями сценария~1,
но подтверждают, что минимальный doc-to-blueprint pipeline уже
работоспособен и~может служить отправной точкой для~улучшения B4.

\subsection{Сценарий~2: Compliance-курс с~факт-чекингом}

\textbf{Входной документ:} корпоративный регламент информационной
безопасности, 22~страницы.

\textbf{Специфика:} документ содержит точные числовые требования
(сроки, пороги, санкции), критичные для~корректного compliance-обучения.

Проверяется, что FactCheckAgent выявляет галлюцинации при~конкретизации
числовых значений. Для~этого уже подготовлен синтетический датасет
из~100 утверждений (\texttt{statements.json}), пригодный для~расчёта
precision / recall / F1. Финальные численные результаты будут
получены после стабилизации B4 и~подключения полного fact-check
контура к~основному benchmark-run.

\subsection{Сценарий~3: Авто-актуализация при~обновлении источника}

\textbf{Условие:} в~исходный документ сценария~1 внесены изменения
(обновлена функциональность продукта, 3~новых раздела).

Проверяется корректность выявления устаревших уроков и~качество
перегенерации затронутых разделов без~перегенерации всего курса.
На~момент подготовки работы сценарий сформулирован как~отдельный
validation-case, однако численная метрика ещё не~зафиксирована.
Это один из~следующих экспериментов, поскольку именно он проверяет
прикладную ценность системы для~B2B-контента, который быстро устаревает.

\subsection{Проверка гипотез H1--H3}

\textbf{H1 (продуктовая).}
«Мультиагентный конвейер сокращает время создания структуры курса
с~X~часов до~Y~минут при~сохранении Bloom's-выравненности $\geq 70\%$.»

На~момент текущего черновика гипотеза получает
\textbf{предварительную поддержку}, но не может считаться полностью
подтверждённой. С одной стороны, проблема времени разработки надёжно
валидирована через 20 интервью, а текущий baseline показывает, что
doc-to-blueprint генерация уже занимает секунды, а не часы. С другой
стороны, полноценный B4 end-to-end сценарий с~human-review и~итоговым
измерением time-to-draft на~реальном пилоте ещё не стабилизирован.
\textbf{Гипотеза H1 предварительно поддержана и требует финального pilot-run.}

\textbf{H2 (техническая).}
«Мультиагентная архитектура обеспечивает более высокое качество структуры
по~сравнению с~single-prompt baseline на~той же LLM.»

Текущее состояние данных позволяет утверждать лишь то, что one-shot
baseline B2 недостаточен по~Bloom coverage и~совсем не обеспечивает
citation-aware поведения. Это создаёт содержательное основание для
мультиагентной архитектуры, но ещё не даёт статистически завершённого
сравнения B2 и~B4. До завершения стабильного benchmark-run гипотеза H2
должна трактоваться как \textbf{не полностью проверенная}.
\textbf{Гипотеза H2 концептуально обоснована, но эмпирически пока не завершена.}

\textbf{H3 (бизнесовая).}
«Unit-стоимость генерации курса $\leq Z\%$ от~стоимости ручной разработки
у~внешнего подрядчика.»

В~сценарной финансовой модели unit-cost генерации курса составляет
75~руб., что эквивалентно примерно 0.05\,\% от~ориентировочной стоимости
ручной разработки курса у~внешнего подрядчика.
Такой результат не доказывает рынок сам по~себе, но показывает, что
архитектура не разрушает экономику продукта уже на~раннем этапе.
\textbf{Гипотеза H3 предварительно поддержана сценарной unit-экономикой.}

\subsection{Производительность}

На~текущем этапе доступны следующие подтверждённые операционные показатели:
\begin{itemize}
  \item среднее время baseline B2 на~ядровом корпусе из~16 документов ---
    4.31~с на~генерацию blueprint;
  \item среднее число модулей --- 3, среднее число уроков --- 6.2;
  \item valid structure rate --- 100\,\%.
\end{itemize}

Нодовая телеметрия p50 / p95, throughput и~GPU-utilization включены
в~план следующей итерации наблюдаемости. Они будут сниматься после
стабилизации B4 через LangSmith / системные метрики рантайма.

\subsection{Точность факт-чекинга}

На~синтетическом датасете из~100 утверждений (50 корректных,
50 намеренно искажённых) уже подготовлен воспроизводимый контур
оценки precision / recall / F1. Численные значения не включаются
в~текущую редакцию как итоговый результат, поскольку полный B4
fact-check pipeline ещё не завершил стабильный прогон.

% ─────────────────────────────────────────────────────────────────
\section{Ограничения исследования и угрозы валидности}
\label{sec:limitations}
% ─────────────────────────────────────────────────────────────────

Настоящий раздел явно перечисляет ограничения исследования и~угрозы
внутренней и~внешней валидности~\cite{wohlin_experimentation_2012}.
Осознание этих ограничений необходимо для~корректной интерпретации
результатов и~определения направлений дальнейших исследований.

\subsection{Ограничения тестового датасета}

\textbf{Ограниченный объём измеренного корпуса~($n = 16$).}
Ядровой корпус содержит 16~документов, что ограничивает
статистическую мощность сравнений.
При~размере эффекта $d = 0.5$ (средний) и~$\alpha = 0.05$
мощность критерия Вилкоксона для~такой выборки остаётся ограниченной
и~не~гарантирует надёжного сравнения тонких различий между конфигурациями.
Следовательно, результаты следует рассматривать как~предварительные
и~требующие репликации на~расширенном корпусе ($n \geq 30$).

\textbf{Доменная специфика.}
Все~15~документов корпуса принадлежат российскому B2B-контексту
(SaaS-продукты, корпоративные регламенты, учебные программы~ФГОС).
Обобщаемость результатов на~другие домены (медицина, юриспруденция,
технические науки) и~другие языки не~подтверждена.

\subsection{Ограничения эталонного корпуса (ground truth)}

На~момент текущей редакции полностью завершённым является не~экспертный
ground truth, а~корпус входных документов и~оценочные наборы
для~RAG / fact-check. Экспертная разметка и~расчёт inter-rater
agreement подготовлены как~следующий шаг, но ещё не могут считаться
окончательно зафиксированными результатами. Это ограничивает силу
выводов о~педагогическом превосходстве B4 по~отношению к~другим
конфигурациям.

\subsection{Ограничения пользовательской валидации}

Пользовательский pilot-контур спроектирован, но ещё не завершён как
формальный исследовательский этап. Поэтому текущая работа пока не даёт
численно подтверждённых SUS / NPS и~не может делать сильные выводы
о~поведенческом удержании, повторном использовании или~организационном
масштабировании. Следующий шаг --- проведение решенческих интервью,
LOI-stage и~пилотов на~реальных документах заказчиков.

\subsection{Ограничения технологического стека}

\textbf{Зависимость от~одной LLM.}
Все~эксперименты проведены на~Qwen~2.5-72B.
Результаты могут не~воспроизводиться на~других моделях
(LLaMA~3.3, Mistral~Large, GPT-4o), что ограничивает
обобщаемость вывода об~эффективности агентной архитектуры.

\textbf{Ограниченный набор методологий.}
На~момент валидации реализована одна методология (ADDIE).
Сравнение с~SAM и~Backward~Design отложено до~следующей итерации
и~не~включено в~основные результаты работы.

\textbf{Авто-классификация Bloom's уровней.}
Уровни таксономии Блума определялись автоматически с~помощью
GPT-4o в~роли~judge-модели~\cite{mt_bench_2023}.
Надёжность такой классификации на~русскоязычных учебных целях
не~валидирована независимо; это является угрозой валидности
измерения Bloom's~coverage.

\subsection{Угрозы внешней валидности}

Система разработана и~протестирована исключительно на~русскоязычном
контенте B2B-сегмента.
Перенос результатов на~другие языки, другие модальности обучения
(офлайн, blended) или~другой регуляторный контекст (SCORM, xAPI)
требует дополнительных исследований.

\subsection{Направления для устранения ограничений}

В~качестве следующих шагов определены:
\begin{enumerate}
  \item расширение тестового корпуса до~30--50~документов
    из~5+~различных доменов;
  \item добавление второго эксперта и~расчёт Cohen's~$\kappa$
    для~оценки надёжности разметки Bloom's уровней;
  \item кросс-модельная оценка: воспроизведение экспериментов
    на~LLaMA~3.3~70B и~GPT-4o для~проверки обобщаемости;
  \item расширенный пилот ($n \geq 20$) с~longitudinal
    метриками удержания контента.
\end{enumerate}

\subsection{Unit-экономика}

Себестоимость одного сгенерированного курса складывается из:
GPU-времени (Qwen~2.5 + FLUX + Silero) и~опциональных API-вызовов.
В~текущей редакции используется сценарный расчёт, собранный
в~отдельном экспериментальном артефакте \texttt{unit\_economics\_results.json}.

\begin{itemize}
  \item GPU-время~--- 75~руб./курс;
  \item OpenAI API (опционально)~--- 0 в~base-конфигурации;
  \item \textbf{Итого unit-cost}~--- $\approx$~75~руб./курс.
\end{itemize}

Ключевые метрики сценарной подписочной модели:

\begin{itemize}
  \item CAC (стоимость привлечения клиента): около 30\,000~руб.;
  \item LTV (lifetime value): около 300\,000~руб.;
  \item LTV / CAC: 10.0;
  \item Payback Period: 2~мес.
\end{itemize}

\subsection{Lean Canvas}

Lean~Canvas~\cite{maurya_lean_canvas_2012} ИИ-методиста приведён
в~приложении~К. Ключевые блоки:

\textbf{Проблема}: разработка корпоративного курса требует значительного
времени и~часто выносится на~внешних подрядчиков; контент устаревает
быстрее, чем его успевают методически перерабатывать.

\textbf{Решение}: автоматическая генерация структуры и~текстов курса
по~загруженному документу; human-in-the-loop ревью; механизм
последующей авто-актуализации.

\textbf{Целевая аудитория}: B2B-клиенты платформы Adapstory~---
HR-отделы и~учебные центры с~>~50~сотрудников.

\textbf{Unfair~Advantage}: методологическая вариативность (ADDIE/SAM/BD)
и~полный суверенитет данных (on-premise LLM) не~воспроизводятся
Coursebox~AI и~iSpring~AI.

\subsection{Потенциал масштабирования}

После~валидации в~российском B2B-сегменте идентифицированы
смежные рынки:

\begin{itemize}
  \item \textbf{Международный EdTech}: потенциальный TAM $\approx$~\$51~млрд
    к~2030~году~\cite{edtech_market_report_2024};
  \item \textbf{Корпоративное обучение (CLO-сегмент)}: интеграция

  \item \textbf{Государственное образование}: адаптация под~ФГОС-регламент
    при~поддержке профильных ведомств.
\end{itemize}

\subsection{PESTLE-анализ при масштабировании}

\begin{itemize}
  \item \textbf{P (Political)}: регуляторные требования к~использованию
    AI в~государственном образовании; ограничения экспорта технологий;
  \item \textbf{E (Economic)}: зависимость от~стоимости GPU-аренды;
    волатильность курсов валют при~API-зависимости;
  \item \textbf{S (Social)}: опасения преподавателей о~замещении;
    требования к~прозрачности AI-генерации;
  \item \textbf{T (Technological)}: быстрое устаревание LLM;
    риск лицензионных изменений в~open-source моделях;
  \item \textbf{L (Legal)}: 152-ФЗ~\cite{fz_152_2006} (персональные данные);
    авторское право на~AI-сгенерированный контент (ГК~РФ~ч.~IV);
  \item \textbf{E (Environmental)}: энергопотребление GPU-кластера;
    углеродный след at-scale.
\end{itemize}

служебное произведение).

\subsection{Текущий статус MVP}

По~состоянию на~момент написания работы реализованы:
Document~Ingestion~Pipeline, CourseStructureAgent (ADDIE), human-review,
ContentCreatorAgent с~RAG, FactCheckAgent.

В~разработке: SAM / Backward~Design методологии, MediaGenerationAgent
(FLUX~+~Silero), механизм авто-актуализации.

\subsection{Traction}

\begin{itemize}
  \item Проведены 20~B2B-интервью (Customer Development);
  \item 70\,\% респондентов называют time-to-development главным pain;
  \item 50\,\% уже используют подрядчиков или~внешних методистов;
  \item 60\,\% готовы рассмотреть AI-инструмент такого класса;
  \item решенческие интервью, LOI-stage и~активные пилоты относятся
    к~следующей волне коммерческой валидации.
\end{itemize}

\subsection{Problem/Solution Fit}

На~момент текущего черновика корректнее говорить не о~полностью
подтверждённом Problem/Solution Fit, а~о~сильном problem-signal
и~раннем interest-signal. Проблема уже подтверждена через CustDev,
а~60\,\% респондентов готовы рассмотреть AI-инструмент в~своём
процессе. Полноценный Problem/Solution Fit будет зафиксирован после
серии решенческих интервью на~MVP и~pilot-использования на~реальных
документах компании.

\subsection{Качественная обратная связь}

Даже на~уровне проблемных интервью рынок дал три устойчивых качественных
сигнала:
\begin{itemize}
  \item пользователям важнее быстро получить \textit{первый черновик}
    курса, чем сразу идеальный финальный контент;
  \item для~регламентов и~compliance-сценариев критична привязка к~источнику
    и~возможность проверки фактов;
  \item для~B2B-сегмента существенны суверенитет данных, on-premise
    развёртывание и~human-in-the-loop контроль.
\end{itemize}
